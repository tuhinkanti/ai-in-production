02

What Is Context Rot?

Run an agent long enough and its own history turns against it.
Chapter two. The thing that breaks when you run an agent for a long time is context rot. Let's define it carefully, because the whole solution falls out of understanding it.
AI IN PRODUCTION · CH 2
Context rot10
Context rot

Long-running agents fill up context.

As agents work through a large problem, they keep growing the underlying LLM context — leading to context rot.
Here's the mechanism. As an agent works through a large problem, every step — every file it reads, every tool result, every bit of reasoning — gets appended to the underlying LLM context. The context just keeps growing. The crate overflows. That overflow is what we call context rot.
AI IN PRODUCTION · CH 2
Definitions11
Context is limited

What is context?

· Context

All the tokens the LLM can see

Everything the model can look at to answer a question — the prompt, the history, the files, the tool output.

· Context window

How many tokens it can process

The amount of tokens an LLM can take in and process at a time. It is finite — there is a hard ceiling.

· Context rot

Accuracy decays as it fills

LLMs become less accurate as context length increases — even before the window is full.

Three quick definitions. Context is all the tokens the LLM can see to answer a question. The context window is how many tokens it can process at a time — it's finite. And context rot is the important one: LLMs become less accurate as the context length increases. It's not just about overflowing the window — quality degrades well before that.
AI IN PRODUCTION · CH 2
The degradation12
The degradation

LLMs hallucinate as context grows — making agents unreliable.

Small context → reliable. Growing context → overwhelmed. Large context → hallucinating and broken.
And this is what it looks like in practice. With a small context the agent is reliable — accurate and predictable. As context grows it gets overwhelmed and distorted — answers start drifting. Push to a large context and it breaks: hallucinating, unreliable. So a naive long-running agent on a 450,000-line refactor lands squarely in the red zone.

03

What Is the Ralph Loop?

A surprisingly simple pattern for letting agents run longer.
Chapter three. The Ralph loop. The name sounds fancy; the idea is almost embarrassingly simple — and that's exactly why it works.
AI IN PRODUCTION · CH 3
Autonomous AI development pattern15
Autonomous AI development pattern

The Ralph loop.

Iterate & improve → code & execute → evaluate & learn — then go around again.
At its heart the Ralph loop is a three-step cycle. One: iterate and improve — refine the plan and strategy. Two: code and execute — generate code and run the task. Three: evaluate and learn — analyze the results and gather feedback. Then it goes around again. The trick is in how each turn of the loop is set up — which is the next slide.
AI IN PRODUCTION · CH 3
It's just a loop16
Ralph is a bash loop
while :; do cat PROMPT.md | agent ; done
That's it. Feed the prompt to the agent, let it work, and do it again — fresh, every time.
Geoffrey Huntley · ghuntley.com/loop
And here's the punchline. Ralph, in its purest form, is a bash loop: while true, cat the prompt file into the agent, done. This framing comes from Geoffrey Huntley. The profound part isn't the bash — it's that each iteration is a fresh invocation of the agent. The loop restarts the context every single time. Hold that thought.

How Does It Work?

Small verifiable tasks, fresh context each time, learnings carried between runs.
Chapter four. Let's make this concrete — what does the loop actually consume, and why does fresh context defeat context rot?

The work list18
prd.json · small verifiable tasks for an agent to complete

Break the refactor into small, verifiable tasks.

github.com/tuhinkanti/developerweek2026-ralph-demo
{
  "id": "US-003",
  "title": "Add conversion methods to LegacyTargetCode",

Each task is tightly scoped — one unit of work an agent can finish in a single pass.
Each carries acceptance criteria the agent can check itself against.
"passes": false until it's objectively done — verifiable, not vibes.
The fuel for the loop is a backlog of small, verifiable tasks — I kept mine in a prd.json. Each entry is one tightly-scoped unit of work, like "add conversion methods to LegacyTargetCode," with explicit acceptance criteria, including that existing tests must still pass. And a passes flag that stays false until the task is objectively done. Verifiable, not vibes. That's the unit the agent picks up each turn.

Why it works19
No context rot

Every iteration has fresh context.

  1. Each task starts with fresh context — no rot carried over.

  2. Verification is self-contained within the task.

  3. Learnings are shared between runs — through files, not the window.

  4. Changes stay small and targeted.

… this pattern is not very different from classic problem solving — divide, conquer, write it down.
And here's why this beats context rot. Every iteration starts with fresh context — the agent reloads the prompt and the task, so nothing rots. Verification is self-contained inside each task. Learnings carry between runs through files on disk, not through an ever-growing context window. And every change stays small and targeted. Honestly, this isn't far from classic problem-solving: divide and conquer, and write down what you learned.

For the loop to actually run unattended, the agent can't stop and ask for permission on every file edit or command. So you run it in what the tools call yolo mode — dangerously-skip-permissions. The name is a warning, and it's earned: you only do this inside guardrails. Which is exactly what the next few slides are — the guardrails that make an uninterrupted run safe.
AI IN PRODUCTION · CH 5
Guardrail · regression23
Commit-validate loop · verify changes do not introduce regression

Every task ships through the same gates.

  1. Agent implements the task from prd.json.

  2. Agent adds unit tests for additional coverage (from the acceptance criteria).

  3. Agent runs a full build and test — monorepo and Bazel, in my case.

  4. Agent pushes a commit, opens a PR, and triggers a CI build.

  5. Agent waits for CI to pass before progressing to the next task.

… ready for human review.
The core guardrail is a commit-validate loop, built to catch regressions. For each task the agent implements the change from prd.json, adds unit tests for the coverage its acceptance criteria demand, runs a full build and test — for me that's a Bazel monorepo — then pushes a commit, opens a PR, and triggers CI. Crucially, it waits for CI to go green before moving to the next task. If CI fails, that becomes the next thing it fixes. Only then is it ready for human review.

Guardrail · review24
AI code reviews · make code review effective for humans

Human review is the quality gate — so make it count.

  1. Automated code reviews using Prizm.1

  2. Reviewed with a different thinking model — Opus 4.5 / GPT-5.3-high.

  3. Iterate until all potential issues are addressed.

… ready for human review.
1 engineering.salesforce.com/scaling-code-reviews-adapting-to-a-surge-in-ai-generated-code
Human review is the real quality gate — but a human shouldn't be the first to read raw agent output. So before it reaches me, the PR goes through automated AI code review with a tool we call Prizm, and importantly with a different thinking model than the one that wrote the code — Opus 4.5 or GPT-5.3-high. A reviewer that thinks differently from the author catches more. The agent iterates against that review until the issues are addressed. Then, and only then, it's ready for a human.

Key takeaways27
Key takeaways · Ralph loop works through context management

What I'd carry forward.

01 · fresh context
LLM agents are very accurate with fresh context.
Reset the window each turn — don't let history rot.
02 · small verifiable tasks
Agent loops accurately solve small, verifiable tasks.
Scope tight, define done, let the loop grind.
03 · the plan is everything
Plan is the most important step in the agent SDLC.
Plan with an expensive model (Opus, GPT-5.x-high). Implement with a cheaper one (Sonnet 4.5, Composer 1, GPT-Codex, Gemini Flash).
04 · SDLC is a prerequisite
Code reviews, test coverage, and CI are prerequisites.
Best practices aren't optional — they're what make autonomy safe.
Four takeaways. One: LLM agents are remarkably accurate when the context is fresh — that's the whole game. Two: agent loops reliably solve small, verifiable tasks, so the work is to break problems down that way. Three: the plan is the single most important step — plan with an expensive, strong model like Opus or a high-reasoning GPT, and you can implement with cheaper models like Sonnet 4.5, Composer 1, GPT-Codex, or Gemini Flash. And four: this only works on top of real SDLC discipline — code reviews, test coverage, and CI aren't nice-to-haves, they're the prerequisites that make autonomy safe.

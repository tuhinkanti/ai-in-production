These Helm charts live in thousands of repositories, and within each one the CPU request can hide in six different config types: Structured Config overrides, helm.tf addon, the chart's own values.yaml, env-specific overrides, Terraform locals, and hardcoded templates. The value you think you're changing often isn't the value that wins. If it's a hardcoded template, no override can save you — you have to edit the template itself. That's what made this risky and tedious to do by hand, at scale.

Two-word definition13
So we stopped tuning the model
A slot machine isn't a model problem. So what was I actually building?
Agent = Model + Harness

I'd been treating the slot machine as a model problem — reach for a bigger model, a bigger prompt, more sub-agents. But none of those were the issue. So I took a step back and asked what an agent actually is. Here's the two-word definition I landed on: an agent is the model plus the harness. Everything that isn't the model is the harness — and those three knobs I kept reaching for were all the model. The harness was everything around it, and that's where the leverage actually was.

Two Kinds of Work.
Back to the bug.
Vocabulary loaded. Now back to the bug: two runs, same input, different PRs. The missing harness piece was hiding in the optimizer sub-agent — the part we had casually written a prompt on how it has to optimize the CPU. It wasn't a very simple thought as we will see shortly; our pipeline was mixing two very different kinds of work.
AI IN PRODUCTION · CH 3
Two buckets15
The split that matters

Every step of an agent pipeline is one of two kinds of work.

Bucket 01
Reasoning
Discover how CPU is configured across this repo.
  • Inputs ambiguous. No single correct path.
  • Tolerance for ambiguity is the LLM's superpower.
  • The agent has to adapt.
Want the creative writer here.
Bucket 02
Structured
Compute the optimal CPU plan across all instances.
  • Inputs well-defined. Constraints known.
  • Answer is mathematically derivable.
  • Same input → same answer, every time.
This is not creative writing.
Every task inside an agent pipeline falls into one of two buckets. Reasoning work is repo discovery: given a repo shape the agent may never have seen, infer how CPU is configured across instances, scopes, and override layers. Tolerance for ambiguity is a genuine superpower of LLMs. Structured work is the opposite: the inputs and constraints are defined, and the answer should be mathematically derivable.
AI IN PRODUCTION · CH 3
The pivot16

We re-architected the middle. Fewer moving parts, a stronger spine.

Reasoning → Structured → Reasoning
So v3 stopped asking nine sub-agents to reason their way to a structured answer. We kept the LLM where ambiguity was real: Discover maps how CPU is configured across instances in that repo, and Apply makes the surgical YAML edits. In the middle, Plan became pure code — a deterministic solver — so the core planner is now 100% repeatable. I still say only the core planner, not whole PR, and we'll come back to that caveat when we open the harness.

This starts with the familiar move: one global default, cranked high enough to cover the worst case. It covers twelve out of twelve, but look at the cost — most instances are overprovisioned. ILP turns this into decision variables: which knobs should be enabled, what value should each knob take, and which instances does each knob cover? Intuitively, the solver is asking: can I cover every instance, waste the least CPU, and use the fewest knobs? A broad knob is cheap to configure but can waste a lot; a specific knob is precise but adds another override to maintain. ILP searches that tradeoff directly. Run it here, and it throws away the global knob, picks PROD, DEV, and one most-specific override for the rebel — three knobs, everything green, same answer every run.

Reasoning on the ends, structured work in the middle — each tool doing what it's best at. That's the context engineering that made the plan stop drifting. And it's safe by construction: the agent only ever tunes CPU requests and never touches CPU limits, so scaling headroom is preserved, and every change ships as a reviewable pull request through the normal pipeline.

Quick recap before we close. Three threads. The agent — v1 was one ReAct loop with a bloated prompt, v2 was nine sub-agents that drifted, and v3 is Discover, Solver, Apply: reasoning on the ends, structured work in the middle. The reframe — Agent equals Model plus Harness; the fix was never a bigger prompt or a better model, it was moving the structured work out of the LLM and into a deterministic solver. And the three lessons to take home: separate reasoning from computation, use deterministic systems for deterministic problems, and improve the harness, not the prompt. The payoff: the plan stopped drifting. Same input, same plan, every run — accuracy that climbs toward 100%. That agent now runs in production across thousands of services, and it's reclaimed millions in capacity. Best practice is not more prompting. It is better harness.
